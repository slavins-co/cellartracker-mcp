# Docs Sync (#53) + Tool Annotations (#54 partial) Implementation Plan

**Goal:** Fix stale skill/README docs to match actual registered tools (#53, full), and add MCP tool annotations (readOnlyHint/openWorldHint/destructiveHint + human titles) to every tool in `src/server.ts` using the SDK's existing overload (#54, annotations slice only).

**Architecture:** Two independent, additive changes in one PR. (a) Markdown-only edits across 4 doc files, zero code impact. (b) A 4th `annotations` argument inserted into each existing `server.tool(name, description, schema, cb)` call in `src/server.ts` — SDK 1.29's `tool<Args>(name, description, paramsSchemaOrAnnotations, annotations, cb)` overload (still supported, deprecated in favor of `registerTool` but functionally identical) already accepts this shape, so no migration to `registerTool` is needed. A new test uses the SDK's own `Client` + `InMemoryTransport.createLinkedPair()` public test harness (confirmed present in `@modelcontextprotocol/sdk` 1.29) to call `listTools()` and assert each tool's `annotations` object — no mocking of CellarTracker network calls needed since listing tools never invokes a handler.

**Investigated & resolved before writing this plan:**
- SDK 1.29.0 `tool()` overload signature confirmed via `node_modules/@modelcontextprotocol/sdk/dist/cjs/server/mcp.d.ts` — supports `(name, description, paramsSchema, annotations, cb)`.
- `ToolAnnotations` fields confirmed via SDK `types.d.ts`: `title?`, `readOnlyHint?`, `destructiveHint?`, `idempotentHint?`, `openWorldHint?`.
- README wishlist wording (#53 item 4): `Tag` table (wishlist) schema has no score columns at all (`references/schema.md:272-291`), and `Availability` (the only table with pro scores + maturity) is documented as "one row per wine **in inventory**" (`references/schema.md:220-222`) — it does not cover wishlist-only wines. Cross-referencing is **not viable**; fixing the README wording is the correct path per the issue's own fallback instruction.
- Doc drift extends beyond the one skill file named in the issue: `skills/wine-purchase-evaluator/SKILL.md:55` and `skills/wine-purchase-evaluator/references/guide.md:31` also reference `search_cellar`/`cellar_stats`/`get_wishlist`/`purchase_history` with underscores. Same bug class as the issue's item 1, in scope per the AC ("skill docs match registered tool names") — fixing alongside.
- `schema.md` has no tool-name or charset drift; no changes needed there.

**Judgment calls (flagging, not blocking):**
1. Per the user's #54 spec, "plus a human-friendly title" is listed only for the 9 data-query tools, not for `setup-credentials`/`clear-user-data`. Following that literally — titles added only to the 9 data tools, annotations-only (no title) for the other two. Easy to extend later if inconsistent titling bugs you.
2. `clear-user-data` gets `readOnlyHint: false, destructiveHint: true` only (no `openWorldHint`) since it's a local-only file deletion — matches the user's spec, which didn't mention `openWorldHint` for it.

- [ ] Step 1: Write the failing test in `src/__tests__/server.test.ts`:
  ```ts
  import { Client } from "@modelcontextprotocol/sdk/client/index.js";
  import { InMemoryTransport } from "@modelcontextprotocol/sdk/inMemory.js";
  import { createServer } from "../server.js";

  describe("tool annotations", () => {
    async function listToolsLive() {
      const server = createServer();
      const client = new Client({ name: "test-client", version: "0.0.0" });
      const [clientTransport, serverTransport] = InMemoryTransport.createLinkedPair();
      await Promise.all([client.connect(clientTransport), server.connect(serverTransport)]);
      const { tools } = await client.listTools();
      await client.close();
      return tools;
    }

    it("marks all data-query tools read-only and open-world", async () => {
      const tools = await listToolsLive();
      const dataTools = [
        "search-cellar", "drinking-recommendations", "cellar-stats",
        "purchase-history", "get-wishlist", "consumption-history",
        "tasting-notes", "recent-deliveries", "refresh-data",
      ];
      for (const name of dataTools) {
        const tool = tools.find((t) => t.name === name);
        expect(tool, `${name} should be registered`).toBeDefined();
        expect(tool!.annotations?.readOnlyHint).toBe(true);
        expect(tool!.annotations?.openWorldHint).toBe(true);
        expect(tool!.annotations?.title).toBeTruthy();
      }
    });

    it("marks setup-credentials as a write, network-calling tool", async () => {
      const tools = await listToolsLive();
      const tool = tools.find((t) => t.name === "setup-credentials");
      expect(tool!.annotations?.readOnlyHint).toBe(false);
      expect(tool!.annotations?.openWorldHint).toBe(true);
    });

    it("marks clear-user-data as destructive and non-read-only", async () => {
      const tools = await listToolsLive();
      const tool = tools.find((t) => t.name === "clear-user-data");
      expect(tool!.annotations?.readOnlyHint).toBe(false);
      expect(tool!.annotations?.destructiveHint).toBe(true);
    });
  });
  ```
  (`setup-credentials`/`clear-user-data` are conditionally registered only when env credentials are absent — same as today's runtime; the test process shouldn't have `CT_USERNAME`/`CT_PASSWORD` set, matching CI. If it does, gracefully skip those two assertions.)
