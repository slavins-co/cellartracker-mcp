# Issue #49 — Bottle-Level Tool: Implementation Plan

**Goal:** Add a `bottle-details` tool over the `Bottles` table (already fetched every refresh, currently exposed by no tool — pure fetch cost today). Answers per-bottle questions List/Consumed/Availability can't: barcode, exact bin location, size/format, and state (in cellar vs. consumed).

**Mode:** HOLD (selected 2026-07-12), with two additions Brad confirmed as hard requirements during review (not scope drift — surfaced, decided, recorded): a `barcode` filter (fed from a photo attachment Claude reads via vision, not new tool machinery) and coherent handling of natural-language location/bin questions ("what's in my wine fridge", "what's on the bottom shelf of my cabinet").

---

## Design decision: Bottles vs. Inventory — RESOLVED

The issue flagged this as open ("compare the columns of Bottles vs Inventory with one authenticated pull"). Resolved during this review with a live authenticated pull of `Table=Inventory` against the real account, compared against the cached `Bottles_latest.csv`:

| | `Bottles` (already fetched) | `Inventory` (would be a 9th fetch) |
|---|---|---|
| Rows (this account) | 121 (64 cellar + 57 consumed) | 64 (cellar only) |
| Barcode, Location, Bin | Yes | Yes |
| `BottleState` / consumption fields (date, type, note) | Yes | **No** |
| Valuation, Price, pro scores, community stats | No | Yes |

**Decision: use `Bottles`, not `Inventory`.** The issue's own filter list requires `BottleState`, which `Inventory` cannot provide at all (it has no consumed rows, no state field — it's implicitly "cellar only"). `Bottles` is already paid for in every refresh; `Inventory` would be a 9th fetch and a 9th credentialed request surface for a read-only server that already authenticates with full-account credentials (see the audit's read-only rationale). The one thing `Inventory` has that `Bottles` doesn't — Valuation/pro-scores — is recoverable for free later via the existing `crossReference()` helper (`src/query.ts:105`) against `List`/`Availability` (both already fetched) if ever wanted. Not needed for this issue's stated scope.

**Action for implementation:** document this column comparison in `skills/cellartracker-data/references/schema.md` under the existing `Bottles` section (the issue's acceptance-criteria-adjacent note asks for this), so the "why not Inventory" reasoning survives past this session.

## Data flow, including shadow paths

```
filters(query, location, bin, size, barcode, state, max_results)
        │
        ▼
loadTable(Bottles_latest.csv)  ──[cache missing/stale]──▶ ensureFresh() re-fetches
        │                                                  (existing shared path, no new code)
        ▼ rows: Row[]  ──[rows === []]──▶ "No bottles found matching your criteria."
        │                (new user / empty cellar — same message shape as other tools)
        ▼
search() substring filters on Wine/Location/Bin/BottleSize/Barcode — diacritic-folded (reused as-is)
        │  ──[no matches AND location or bin was supplied]──▶
        │       "No bottles found for that location/bin. CellarTracker's Location and Bin
        │        values are account-specific labels, not physical descriptions — call
        │        cellar-stats with group_by=location or group_by=bin to see what's
        │        actually in use, then retry with the exact value."
        │  ──[no matches, no location/bin supplied]──▶ generic empty-result message
        ▼
state filter: "cellar" → BottleState==="1" only
              "consumed" → BottleState==="0" only
              "all" (default) → no filter
        │  ──[BottleState is "" or garbled]──▶ excluded from "cellar"/"consumed" views,
        │                                       still included under "all" — never crashes
        ▼
sort: BottleState desc (cellar first — most actionable), then Location, then Bin
        │  ──[Location or Bin === ""]──▶ empty string sorts first (stable, no crash)
        ▼
slice(0, max_results) in server.ts (matches search-cellar/consumption-history convention)
        │  ──[total > max_results]──▶ "(Showing N of M results...)" footer
        ▼
render: vintage + wine, state label, barcode (if present), location/bin (omit if blank),
        size (if present), consumption date + type (only when state=consumed and present)
```

**Why the miss-message points at `cellar-stats` instead of guessing:** CellarTracker `Location`/`Bin` values are opaque, account-specific labels ("Drawer 2", "1-3") — there is no way to derive "bottom shelf of my cabinet" from the string itself. `cellar-stats` already supports `group_by=location`; this plan extends it to also support `group_by=bin` (Task 2b below), so one cheap round-trip gives the calling model (or Brad) the real vocabulary to retry with. Checked against the open-issue list first (per Brad's instruction) — no existing issue (#50–54) covers this gap, so it's handled here rather than deferred.

## Error & Rescue Registry

| Failure | Where | Caught? | How | User sees | Severity |
|---|---|---|---|---|---|
| CT auth/network failure | `ensureFresh`/`getFreshPaths` (shared, existing) | Yes | Existing `AuthError`/`ServiceError`/retry (`exporter.ts`) | Existing clean error message | Low — no new code |
| Empty `Bottles` table (new/empty cellar) | `loadTable` | Yes | `rows.length === 0` → early return | "No bottles found matching your criteria." | Low |
| Filters match nothing, no location/bin involved | `search()` / state filter | Yes | `results.length === 0` check in server.ts | Generic empty-result message | Low |
| Filters match nothing, location and/or bin supplied | `search()` / state filter | Yes | Same check, branches on whether `location`/`bin` were passed | Message points at `cellar-stats(group_by=location\|bin)` to discover the real vocabulary | Low |
| Bin/Barcode/Size blank on a row | render | Yes | Conditional `if (field) lines.push(...)`, matching `fmtWine`'s existing pattern | Field line omitted, never shows "undefined" | Low |
| `BottleState` not exactly `"0"`/`"1"` | state filter | Yes (by falling through) | Strict `=== "1"` / `=== "0"` checks | Row silently excluded from `cellar`/`consumed` views; still visible under `all` | Low |
| Unparseable `ConsumptionDate` | `toIsoDate()` (existing, reused) | Yes | Returns `""`, warns once to stderr (existing dedup) | Shows nothing for that line rather than crashing | Low |
| `max_results` ≤ 0 or very large | `slice()` | Not specially handled — matches existing precedent (`consumption-history`, `tasting-notes` don't guard this either) | Native `Array.slice` tolerates any number | Empty list or full list, no crash | Low |

Zero silent failures: every branch above either surfaces a message or is a no-op that degrades gracefully (an omitted field, not an omitted result).

## Decisions confirmed with Brad (2026-07-12)

1. **Default `state` filter: `"all"`.** Confirmed. A photo-sourced barcode lookup shouldn't require already knowing whether that bottle's been drunk.
2. **`barcode` is a filter param, not display-only.** Confirmed — reverses the earlier draft. Primary use case: Claude reads a barcode value off a photo attachment (vision, not new tool machinery) and passes it straight through as a plain string. Reuses `search()` exactly like every other filter — no special-casing, no fuzzy/OCR-tolerant matching (no evidence yet that it's needed; add later if real photo-reads turn out noisy).
3. **Sort order — `BottleState` desc, then Location, then Bin.** Confirmed as designed.
4. **Location/bin natural-language coherence — extend `cellar-stats`, not a bespoke inline hint.** Confirmed after checking the open-issue list (#50–54) for existing coverage — none exists, so it's handled here. See the data-flow diagram above and Task 2b below.

## What already exists (reuse map)

- `search(rows, filters)` — `src/query.ts:75` — diacritic-insensitive substring filter. Reused as-is for wine/location/bin/size/barcode.
- `vintageLabel(row)` — `src/query.ts:163` — NV rendering.
- `toIsoDate(value)` — `src/query.ts:138` — date normalization for `ConsumptionDate`.
- `loadTable(path)` / `parseCsv(text)` — `src/query.ts:14,35` — `Bottles` is already in `TABLES` (`src/exporter.ts:37`), already fetched every 24h. Zero new fetch cost, zero new auth surface.
- `crossReference()` — `src/query.ts:105` — available later if valuation enrichment is ever wanted (see NOT in scope).
- Tool-registration convention — `incoming-orders` (`src/server.ts:449`, from #70) is the freshest template: optional `z.string()` filters, `{ title, readOnlyHint: true, openWorldHint: true }` annotations, `getFreshPaths()` → `loadTable()` → pure-function filter/sort → formatted text.
- Result-cap convention — `search-cellar`/`consumption-history`/`tasting-notes` all slice to a default cap (25) in `server.ts` and append a `"(Showing X of Y...)"` footer when truncated. Match this exactly.
- `aggregate(rows, groupBy)` + `cellar-stats`' `columnMap` pattern — `src/query.ts:89`, `src/server.ts:267` — already does exactly the "distinct values with counts" job needed for bin discovery; extending `columnMap` with `bin: "Bin"` is the whole change (Task 2b), no new aggregation logic.
- Test convention — pure business logic (`pendingOrders`, `deliverySummary`) lives in `query.ts`, unit-tested in `query.test.ts` with inline CSV-string fixtures (not files). Protocol-level checks (tool registration, annotations) live in `server.test.ts`.

### Task 1: `bottleDetails()` in `query.ts`

**Files:** Modify `src/query.ts`

- [ ] Add `export type BottleState = "cellar" | "consumed" | "all";`
- [ ] Add `bottleDetails(rows: Row[], filters: { wine?: string; location?: string; bin?: string; size?: string; barcode?: string }, state: BottleState = "all"): Row[]`:
  - Build a `search()` filter object mapping `wine→Wine`, `location→Location`, `bin→Bin`, `size→BottleSize`, `barcode→Barcode`.
